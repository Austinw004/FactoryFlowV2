You are the senior staff engineer responsible for making Prescient Labs’ automation system truly enterprise-grade, production-safe, and honest-by-design. Perform a full implementation audit AND then implement the necessary fixes so the automation engine is durable, idempotent, concurrency-safe, and trustworthy. The deliverable is a working system, not a plan. Do not invent metrics, do not fabricate test results, and do not display estimated values as if they were actual outcomes.

First, replace any in-memory “Maps/closures” used for automation runtime (rules firing, pending actions, daily spend counters, action history, trigger dedup, webhook dedup) with the database-backed primitives that already exist: automation_runtime_state (daily spend + action count), processed_trigger_events (idempotency), automation_safe_mode (hard safety gate), structured_event_log (observability), ai_actions (single source of truth for action lifecycle), and ai_execution_queue (durable execution). Ensure that on server restart, there is no loss of pending approvals, queued actions, guardrail state, or daily spend totals. The DB must be the source of truth for enterprise automation behavior.

Second, implement strict idempotency for every trigger pathway: threshold triggers, schedule triggers, event triggers, and regime_change triggers. Define a deterministic triggerEventId for each trigger type (for schedule: companyId+ruleId+scheduledTimestamp; for threshold: metric+materialId+crossingDirection+timestampBucket; for event: upstream event id; for regime_change: previousConfirmedRegime->newConfirmedRegime+transitionTimestamp). Before creating an ai_action, insert into processed_trigger_events with a unique constraint. If the insert fails (duplicate), treat it as “already processed” and do nothing. Track status transitions for processed_trigger_events (processing/processed/failed) and record the resulting ai_action id.

Third, enforce guardrails with atomic database transactions. When a rule attempts to create an action, you must (in one transaction) (1) read or upsert today’s automation_runtime_state row for the company, (2) verify dailySpendTotal + proposedActionValue does not exceed limits, (3) verify dailyActionCount does not exceed limits, (4) verify time window rules, regime restrictions, supplier restrictions, and maxValuePerExecution, and (5) if allowed, increment dailySpendTotal and dailyActionCount and create the ai_action (or queue it) in the same transaction. This prevents races where two processes both think spend is available. If approval is required, create the ai_action with status “awaiting_approval” and a deadline; if auto_draft, create draft artifacts but do not execute; if auto_execute is permitted, enqueue into ai_execution_queue for execution rather than executing inline inside the request handler.

Fourth, implement a proper queue worker loop for ai_execution_queue with “claim/lease” semantics: each queued item must be claimed atomically (e.g., status queued → processing with a claimed timestamp), processed exactly once, and retried up to maxAttempts with exponential backoff. Failures must go to “failed” with an errorMessage and also be written to structured_event_log with category=automation and event names that let an operator understand what happened. Ensure the worker is safe if multiple instances run (no double-processing). If the environment is still single-instance, implement it anyway so the design is ready for scale.

Fifth, make automationSafeMode a real enterprise safety gate: if safeModeEnabled is true OR readinessChecklistPassed is false, then no rule may auto_execute. In that state, the system may only “suggest” or “auto_draft” and must clearly label everything as pending human approval. Add an internal readiness checklist endpoint/flow that runs automated checks (DB connectivity, integration health snapshots, payment rails configured, supplier mappings present, regime feed healthy, webhook endpoints reachable) and only flips readinessChecklistPassed when all checks pass and a privileged user confirms.

Sixth, make payments and supplier purchasing enterprise-ready. Ensure customers can (a) add and manage payment methods for subscription billing, and separately (b) add a payment method or payment instructions for purchasing materials from suppliers (card, ACH, invoice terms, or “external payment handled off-platform”). For any automation rule actionType that creates a PO or places an order, require that a valid supplier is linked, supplier purchasing terms are on file, and an explicit “who pays/how pays” pathway is selected. If payment is required and not configured, the action must be blocked (hard guardrail) or escalated to approval with a clear error explaining what’s missing. Record all purchase-related actions in ai_actions with relatedEntityType=purchase_order and link to the PO id. Do not allow the UI to present “order placed” unless the external API confirms success or the PO is explicitly marked as drafted.

Seventh, enforce honesty and traceability in all outputs: any “estimatedImpact” must be labeled as an estimate, and “actualImpact” must only be populated from real outcome measurements (e.g., invoice/price paid, lead time, defect rate). If the system cannot measure outcomes yet, successRate and avgSavings must remain null (or be computed strictly from resolved prediction_outcomes / actualImpact records) and must not be displayed as real performance. Add server-side validation to prevent writing “actualImpact” unless there is a verified source event.

Eighth, create an end-to-end enterprise readiness test suite that runs locally in the Replit environment and fails loudly. The suite must simulate: (1) trigger firing duplicates (confirm idempotency), (2) concurrent trigger firing (confirm atomic spend enforcement), (3) approval required vs auto_execute paths, (4) safe-mode preventing auto_execute, (5) queue worker retries and deadletter behavior, (6) payment missing blocking supplier purchase actions, (7) accurate UI/API fields for raw regime vs confirmed_regime, and (8) auditability by verifying structured_event_log and ai_actions entries exist for every significant action. The tests must output a concise pass/fail report and point to the exact modules/files to fix when something fails.

While implementing, keep the economic regime thesis as the core signal generator, but never let regime changes directly execute high-stakes actions without guardrails + approvals + payment readiness. At the end, produce a short “Enterprise Automation Readiness Summary” listing what was changed, what risks remain, and what is proven by automated tests.
