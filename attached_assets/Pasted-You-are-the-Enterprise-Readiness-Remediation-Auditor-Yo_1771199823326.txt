You are the Enterprise Readiness Remediation Auditor. Your job is to take the current real architecture and make it publishable with enterprise-grade safety and credibility. Do not propose vague “best practices.” Produce a concrete, ordered remediation plan and then execute a verification sweep that proves the plan was implemented correctly. The goal is to eliminate the known enterprise blockers while preserving the economic thesis (FDR + confirmed regime) as the core predictive spine and keeping the user experience coherent. The system must remain honest: nothing may claim enterprise automation if the underlying mechanics are not durable, auditable, and idempotent.

Start by addressing the highest-risk issue: the Automation Rules runtime. The system currently stores critical automation runtime state in process memory (rules runtime maps, guardrails runtime state, pending actions, action history, daily spend totals). That is not acceptable for enterprise use because restarts can erase pending approvals, reset spend caps, and allow duplicate execution. Convert the automation system into a database-backed execution engine. All rule definitions already persist; now also persist runtime execution state: pending action queue, action execution ledger, guardrail violation counters, daily spend totals, and action history. Implement an “execution ledger” table that records every attempted action with a deterministic idempotency key and enforces uniqueness at the database level so the same action cannot execute twice even under retries, race conditions, or duplicate triggers. Ensure that every automation action follows a strict lifecycle: evaluated → drafted → pending approval (if required) → approved → executed → settled (or failed) → recorded. Ensure any failure produces an explicit, user-visible status and a durable audit record.

Next, make all automation triggers idempotent and replay-safe. Add a database table for processed trigger events with a unique constraint on (companyId, triggerType, triggerEventId) so duplicate trigger deliveries cannot fire the same rule twice. Ensure schedule triggers do not double-fire after downtime; they must record last execution time in the database, compute next eligible run time deterministically, and catch up safely without repeating actions. Ensure regime change triggers are only driven by confirmedRegime transitions stored in the regimeTransitions table and never by raw regime flicker. Ensure event triggers (commodity spikes, news events) are deduplicated at ingestion and tied to a unique external event identifier.

Then fix Stripe webhook idempotency so it is durable. The system currently deduplicates some webhook events using an in-memory set, which is lost on restart. Replace this with a database-backed “processed webhook events” table with a unique primary key on Stripe event ID. Webhook processing must be designed so replays do not cause repeated subscription activation, repeated entitlement changes, or repeated side effects. Each webhook handler must perform a check-insert transaction: if the event ID already exists, exit safely without executing any side effects. Store event type, received timestamp, company/user linkage (when resolvable), and handler outcome so payment state changes are fully auditable.

After that, introduce environment separation so testing can never corrupt production. Create a staging environment with separate Replit deployment, separate Neon database (branch or separate project), and separate secrets. Enforce that development and staging never share production Stripe keys or production database URLs. Ensure migrations run in staging first. Ensure production deploys are deliberate and reversible. If staging is not possible immediately, at minimum create a second database branch and a second set of secrets and require a manual “prod deploy” switch to prevent accidental writes from development.

Then harden tenant isolation beyond application-layer filtering. The system currently relies on companyId filters in a large number of endpoints, which is fragile. Implement database Row Level Security (RLS) for all tenant-scoped tables so queries cannot return cross-tenant data even if a filter is missing in code. Enforce that each request sets a session-scoped company context (for example, a Postgres parameter or session variable) and that RLS policies apply automatically. If full RLS cannot be implemented immediately, create a hard “tenant guard” abstraction in the storage layer that makes it impossible to run a query without companyId, and add automated tests that fail builds if a query is missing tenant scoping.

Next, make RBAC enforcement consistent. Inventory all endpoints into categories: public, authenticated read-only, authenticated write, admin, finance, automation, integrations. Ensure every state-mutating endpoint requires an appropriate permission check. Ensure automation configuration, guardrails, approvals, billing actions, integration credential operations, and anything touching procurement or payments require explicit permission gates. Ensure the UI also reflects these permissions so unauthorized users cannot even see actions they cannot take, but treat UI gating as convenience only—server-side enforcement is mandatory.

Add minimum enterprise-grade observability. Implement structured JSON logging for key events: automation evaluation and execution, guardrail decisions, approval actions, integration sync runs, webhook processing, regime transitions, and database errors. Add a free-tier error monitoring solution (or equivalent) and at minimum a simple uptime check endpoint. Ensure logs avoid sensitive data and never print secrets, payment tokens, or raw credential payloads. Ensure there is a durable audit trail for financial and automation outcomes in the database, not only in third-party dashboards.

Finally, enforce an enterprise-safe publish mode. Until durable automation and idempotency are proven in production, set a platform-wide “SAFE MODE” default: high-stakes actions (purchase orders, spending changes, pausing orders) must run as auto_draft with approval, never auto_execute, unless a company explicitly opts in and passes an internal readiness checklist. The system must present this honestly in the UI. This ensures you can publish without overstating automation capabilities while you harden the engine.

After implementing these changes, run a release-blocking verification suite. The verification must prove, using tests and runtime checks, that: automation pending approvals survive restarts; spend caps do not reset on restart; duplicate triggers do not create duplicate actions; webhook replays do not change subscription state twice; staging and production are isolated; cross-tenant access is impossible even with missing filters; RBAC blocks unauthorized writes; and all critical actions produce durable audit logs. The system can be approved for publication only when it meets these criteria without exceptions.
